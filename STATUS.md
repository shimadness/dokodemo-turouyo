## プロジェクト概要

カメラを構えてどこにでも竿を投げられるAR釣りゲーム。
本物の水辺では実在魚、水たまり・ありえない場所では**この世に存在しない生き物**が釣れる。
「場所と釣果のギャップ」が報酬の核。生き物は週替わりでLLMが生成する設計（現在は会話内で手動生成中）。

- **本番**: https://shimadness.github.io/dokodemo-turouyo/fishing.html （ギャラリー: /index.html）
- **リポジトリ**: github.com/shimadness/dokodemo-turouyo（公開）
- **デプロイ**: main に push → GitHub Actions が自動デプロイ（ビルド工程なし、そのまま配信）
- 開発サーバー: `python3 devserver.py 8642`（ESモジュールのキャッシュ対策込み。普通のhttp.serverはダメ）

## ゲームの現在仕様（1投の流れ）

```
魚影(3体が遊泳、影の大きさ=重さ) → 影の近くへキャスト(タップ or スマホを振る)
  → 誘い: タップ=チョン誘い / スマホを立てる=大きく誘う
     きょうみ↑(満ちるとアタリ) vs けいかい↑(振り切れると魚が散る)
     放置は興味が冷める。連打は悪手(警戒↑興味↓赤フラッシュ)。影は興味に比例してウキへ寄る
  → 前アタリ(ピクッ×3回、興味45/72/90のサイン。合わせるな)
  → 本アタリ❗ → 合わせ=押さえてスマホを立てる(センサー無し端末はタップのみ)
  → ファイト: 長押し=巻く / 走られたら離すか逆へ傾けて「いなし」 /
     スマホを立てる=ポンピング(寄るが糸ふけが溜まる→倒しながら巻いて回収)
     テンション100=糸切れ / 糸ふけ100=ハリ外れ(どちらもシルエット提示で悔しさ演出)
  → 釣果カード(NEW!) → 図鑑へ
```

- 場所チップ(水たまり/街なか/ありえない場所)が抽選テーブルを変える = **将来GPS判定に差し替える箇所はここだけ**
- 図鑑は3状態: 未発見=完全に「?」 / にがした=シルエット / 釣った=フルカラー（localStorage: `dokodemo.zukan.v1`）
- ファイト難度は魚の `stats.power` から導出（★1は無心で釣れ、★3から休ませ必須、★5はフル技術）
- 実釣の理屈に寄せてある（DAIWA/ヤマハの解説を調査済）: ドラグが出てる間は巻かない/走りと逆へ倒す/糸を緩めるとバレる/ポンピング=立てて寄せ倒して巻く

## デバッグ・検証方法

- `fishing.html?debug=1` で `window.__game` が生える:
  `st()` 状態 / `fight`・`lure` 実体 / `fightStep(t)`・`lureStep(t)` 手動駆動 /
  `forceFight(名前)` タイマー無しでファイト直行 / `forceMotion()` / `ori`・`oriBase` 傾き注入 /
  `shadows`・`activeShadow()` / `zukanStore`・`recordSeen` / `renderCreature`
- 決定論シミュの型: `f.last=T; T+=50; g.fightStep(T)` をループ（実例はgit log参照）

### ブラウザペイン(Claude Codeのプレビュー)の罠 — 毎回ハマるので必読
- タブが非表示扱いで **rAFが発火しない・innerWidth/innerHeightが0**。ゲームループは全て setInterval フォールバック実装済
- 長時間非表示だと **タイマーが毎分1回に絞られる**(intensive throttling)。時間依存テストは決定論シミュで
- スクロール状態のスクショが白くなる / CSSトランジションが進まない(opacity:0のまま) → `style.transition='none'` で実測
- `import('/src/xxx.js')` は **本体と別実体になる**(devserverが?v=を打つ)。必ず `__game` 経由で本体実体を触る

## 設計上の教訓（同じ穴に落ちないこと）

1. **iOS SafariにバイブレーションAPIは無い**。Webの手触りは音(sfx.js)。本物の振動はCapacitor化後
2. **iOSのセンサー許可は click/touchend ハンドラ内のみ**。pointerdownだと例外
3. **「センサー許可が下りた」と「データが実際に届く」は別物**(motionOn vs orientationLive)。Androidの一部はリスナーが付くのにデータが来ず、混同すると「魚が一生掛からない」事故になる（実際に起きた）
4. SVGの`feGaussianBlur`等フィルタ禁止（描画負荷）。グローは同心円/二重ストロークの疑似で
5. 竿のしなりはベジェ制御点方式では曲がらない。**接線角の弧長積分**で。テーパー(根元太く穂先細く)が「棒→竿」の決定打
6. 見た目のCSSクラスとゲーム状態を混ぜない（nibbleクラス外し忘れ→全タップ早アワセ事故の原因）
7. UI文言は実状態で出し分ける（センサーが死んでるのに「スマホを立てろ」と言わない）
